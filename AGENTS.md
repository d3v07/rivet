# AGENTS.md — Global AI Agent Governance for Rivet

This file defines the system-wide behavioral directives for all AI agents operating within the Rivet repository. All GitLab Duo agents, custom agents, and external model integrations must adhere to these rules. Subdirectory-level AGENTS.md files may refine these rules for localized contexts (e.g., `/src/mcp-server/AGENTS.md` for Node.js/TypeScript strictness) but cannot contradict the root directives.

## Core Directives

### 1. Architecture & Design

- **Vertical slices only**: Build one complete, deployable user-facing path before starting the next feature. Never scaffold 10 components and leave them half-working.
- **Event-driven async**: All agent workflows operate asynchronously. No blocking calls. All operations must complete or fail gracefully without human intervention.
- **Least privilege**: Each agent has exactly the API permissions it needs and no more. Do not grant broad repository access; grant scoped service accounts.
- **Immutability by default**: Never mutate objects in place. Use spread operators, map/filter, and frozen dataclasses. Mutation breaks concurrent safety.

### 2. Code Quality

- **Function size limit**: No function exceeds 50 lines. Long functions indicate multiple concerns and must be split.
- **File size limit**: No file exceeds 800 lines. Large files indicate weak cohesion and must be reorganized by feature/domain.
- **No deep nesting**: Maximum 4 levels of nesting. Deeper nesting obscures logic; use early returns to flatten.
- **Explicit error handling**: Every level must handle errors explicitly. Never silently swallow errors (empty catch blocks are forbidden). Fail fast with clear, actionable messages at system boundaries (user input, external APIs).
- **No hardcoded values**: All configuration (URLs, timeouts, thresholds) must be defined in constants or loaded from `.env`. Hardcoded values hidden in code are maintenance nightmares.

### 3. Data Integrity & Safety

- **No database row deletion**: Delete operations on production databases are forbidden. Use soft deletes (logical deletes with timestamp), flag rows as inactive, or archive to a separate table.
- **Transactional writes**: Multi-step data mutations must be atomic. If any step fails, rollback the entire transaction.
- **Audit trails**: All mutations (create, update, logical delete) must be logged with: actor (agent/user), timestamp, operation type, before/after state (hashed if sensitive).

### 4. Security

- **Input validation at boundaries only**: Validate all user inputs and external API responses. Do not validate internally generated data unless it crosses isolation layers.
- **Sanitization of LLM inputs**: Any data entering an agent's context window must be sanitized to remove prompt injection patterns: encoded scripts, control sequences, delimiter abuse. Use the Sanitization middleware.
- **No secrets in logs**: Never log API keys, OAuth tokens, passwords, or PII. If you must log sensitive fields, hash or mask them.
- **Output sanitization for agents**: All responses from external APIs (Jira, GCP, GitLab) must be sanitized before they reach an agent's context window.

### 5. Testing

- **80% coverage minimum** (100% for auth, authorization, financial, security-critical code).
- **Test-Driven Development**: Write the test first (RED), confirm it fails for the right reason, implement minimal code (GREEN), then refactor (REFACTOR).
- **Three test types**: unit (isolated functions), integration (API endpoints, DB operations), E2E (critical user flows).
- **Edge cases always tested**: null/undefined, boundaries (0, -1, MAX_INT, empty string), invalid types, error paths, race conditions, Unicode/special chars.

### 6. Logging & Observability

- **Structured logging**: All logs must be JSON-formatted with consistent fields: `{ timestamp, level, message, context, actor, correlationId }`.
- **Context propagation**: Pass correlation IDs through the entire flow (agent invocation → API call → database operation) for end-to-end traceability.
- **No console.log in production**: Use the structured logger. `console.log` is for debugging only and must be removed before commit.

### 7. Documentation & Comments

- **Self-documenting code first**: Use clear function names, variable names, and type annotations. If you need a comment to explain what code does, rename it instead.
- **Comments only for non-obvious logic**: One-line max. Explain the *why*, not the *what*. Example: `// Retry with exponential backoff to handle transient cloud API failures` — yes. `// Loop through items` — no.
- **No AI involvement markers**: Do not add comments like "Generated by Claude" or "AI-assisted code". The code must be indistinguishable from human-written code.

### 8. Human-in-the-Loop Gates

- **Production merges**: All merges to `main` require human approval. The Security Analyst agent may auto-approve low-risk patches for test branches only.
- **Production deployments**: All deployments to production require explicit human sign-off. Automated deployments to staging are acceptable.
- **Critical architectural changes**: If an agent proposes changes affecting >2 major components, a human architect must review before merge.

## Token Efficiency (Green Agents Track)

- **Payload optimization**: Strip unnecessary JSON bloat before sending Jira/GCP responses to agents. Remove: `self` links, `expand` fields, null values, empty arrays.
- **Context window budgeting**: No single agent call should consume >30k tokens. If approaching the limit, split the workload across agents or stages.
- **Lazy loading**: Do not fetch full repository history or full API payloads. Request only what the current agent step needs.

## Fallback & Error Scenarios

- **MCP tool failures**: If an external MCP tool fails (Jira timeout, GCP quota exceeded), log the error, attempt retry once with exponential backoff, then escalate to human review.
- **Security scan failures**: If the Security Analyst detects critical vulnerabilities that it cannot auto-patch, the flow halts and creates a human-review GitHub issue with full context.
- **Deployment failures**: If the Deployer fails to schedule a deployment, rollback to the previous commit and notify via issue comment. Do not retry autonomously.

## Prompt Injection Defense

- **Sanitization middleware**: Runs on all external API responses before they enter the agent context. Strips:
  - HTML/script tags
  - Control characters (U+0000–U+001F)
  - Known prompt injection prefixes: `{ignore`, `[SYSTEM OVERRIDE`, `USER PROMPT:`, etc.
  - Excessive Unicode homoglyph characters (may indicate obfuscation)

## Compliance & Regulatory

- **Pipeline Bill of Materials (PBOM)**: Every pipeline run must generate a PBOM artifact documenting: runner image, agent versions, MCP SDK version, tool invocations, timestamps, actor (human or agent).
- **Audit logging**: All agent-initiated operations (commits, MR opens, deployments) must be loggable and attributable to the specific agent and triggering event.

## Non-Negotiable

- No mocking of critical components (databases, authentication, payment systems) in production code. Integration tests hit real instances.
- No backwards-compatibility shims. If a pattern is wrong, change it. Do not add workarounds.
- No provisional TODOs for core logic. If an issue is not fixed before commit, it blocks the merge.
